Optimizing YOLO Video Preprocessing Pipeline via NVMe-oF Based Near-Data Processing

Offloading video decoding, frame sampling, and letterbox resizing to the storage node — delivering only preprocessed frames to the host.

01 Motivation

Smart city and security infrastructure increasingly rely on high-resolution CCTV feeds for real-time object detection. YOLO-based DNN models achieve high inference throughput — yet the preprocessing pipeline dominates end-to-end latency, not inference itself.

In a disaggregated storage environment, the host must first transfer the entire compressed video file over the network before any frame can be decoded or filtered. For a 4K 60fps H.264 stream, the host then decodes every frame and converts it to float32 CHW format — an operation (mat_to_image) that alone accounts for 53% of total pipeline time in Darknet.

This creates two compounding bottlenecks: unnecessary data movement (full video transferred even though only sampled frames are needed) and host CPU saturation from preprocessing that leaves GPU inference starved.

02 Proposed Design

We extend SPDK's NVMe-oF target with two vendor-specific NVMe commands to offload preprocessing to the storage node. The host sends file extents via 0xC0; the storage node decodes, samples, and compresses; the host retrieves JPEG frames via 0xC2 and runs GPU inference.

Baseline (Host)

1 Read compressed MP4 (NVMe-oF)
2 H.264 Decode (OpenCV/FFmpeg)
3 BGR → RGB
4 float32 norm + HWC→CHW 53% bottleneck
5 Letterbox resize (416×416)
6 GPU Inference
→ NDP

NDP (Storage Node)

1 Read MP4 via NVMe SSD (local)
2 H.264 Decode (FFmpeg)
3 Frame Sampling (20%, 60→12 fps)
4 YUV→RGB + Letterbox (416×416)
5 JPEG Compress (quality 85)
6 0xC2 transfer → Host
7 JPEG Decompress + float32 norm
8 GPU Inference host only

The storage node runs an isolated worker thread (pinned to cores 2–15) for FFmpeg decoding, keeping the SPDK reactor free for I/O polling. Results are serialized into a contiguous buffer with a frame-size header table, enabling O(1) frame access on the host.

End-to-end latency (testc_15s, 15s 4K clip): Baseline 81.9s → NDP w/ sampling 5.9s. Speedup is stable across all input lengths (15s–10min) because both pipelines scale linearly with frame count.

Preprocessing breakdown (10min input): Baseline preprocessing accounts for 95.0% of total time (3019.8s of 3178.1s). NDP offloads this to the storage node — the host performs only JPEG decompression and float32 normalization.

Data transfer: NDP with sampling reduces transfer by 92–93% consistently, independent of input length. Without sampling, 58–60% reduction is achieved from JPEG compression alone.
